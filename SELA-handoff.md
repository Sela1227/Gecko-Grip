## 二、發現的「跨專案通用坑」（建議進 Kit）

> Kit 坑庫現有最大編號 **#66**，以下建議從 **#67** 起編。

### 強烈建議加坑

#### 1. localStorage key 是資料層 ID，不隨 App 更名／版本變動

- **症狀**：App 從 `SELA Vocab` 更名為 `Gecko Grip` 時，若順手把 localStorage key 從 `sv3` 改成 `gg1`，使用者累積數月的學習進度全部讀不到，等同從零開始。**不會有任何錯誤訊息**，只會看到「空白的新 App」。
- **原因**：更名是**顯示層**的事，localStorage 以 key 定址是**資料層**的事，兩者無關。但改名時「把所有出現舊名的地方都換掉」是很自然的直覺動作，key 剛好也長得像名字（`sv3` = sela vocab 3），就一起換了。
- **做法**：
  - **storage key 一旦上線就永久凍結**，不隨 App 名稱、版本、品牌調整而變動。key 是識別碼不是文案。
  - 真要改 schema 必須寫 migration：讀舊 key → 轉換 → 寫新 key → 刪舊 key，且保留 fallback 一個版本。
  - 建議在專案 CLAUDE.md「關鍵檔案路徑」表把 key 標成「**唯一真相、不可改**」。
- **影響範圍**：所有用 localStorage / IndexedDB / sessionStorage 的前端專案（Kit 靜態 HTML、React PWA 都適用）。
- **證據**：本專案 CLAUDE.md 坑 P6；V0.4.0 更名時的實際決策點。
- **檢查 1 結果**：grep Kit `localStorage` → 現有 5 處全在講「API key 存 localStorage 不存 DB」的**隱私**議題（#19、sela-philosophy §313），**無任何一條談 key 命名／遷移**。與坑 #4「Schema 演進坑」相鄰但不同：#4 講的是資料結構升版（Dexie/IndexedDB stores），本條講的是 **key 本身被更名帶著跑**。→ **無重複，建議新增 #67**。

#### 2. 「可選的反向模式」要盤點所有進出點，不能只做開啟那一條

- **症狀**：本案卡片主流程是「英文可見 → 遮中文 → 揭示中文」，拼字測試需要的卻是反向的「**遮英文** → 憑記憶輸入 → 揭示英文」。第一版只實作了「開啟拼字 → 遮蔽」，結果使用者按 T 開啟後、不確認就換下一張，下一張的英文仍是遮蔽狀態（狀態殘留）。
- **原因**：任何「可選模式」都會改動共用的 UI 狀態，但開發時的注意力全在**主路徑（開啟）**，逃生路徑（收起 / 中途離開 / 換下一筆 / 直接送出）容易漏。而且這類 bug **只在「開了但不走完」時才出現**，happy path 測不到。
- **做法**：
  - 實作可選模式時**先列出所有進出點再寫程式**。本案是四個：開啟 → mask、確認 → unmask、收起 → unmask、換下一張 → unmask。
  - **狀態用獨立節點承載，不要改共用節點的內容**。本案用獨立的 `#en-mask` 承載遮罩、`#en-word` 只切 `.hidden`；若改 `#en-word.textContent`，還原時要重存原字串，多一個掉字串的機會。
  - 煙霧測試必須包含「**開了但不走完**」的路徑，不是只測「開了走完」。
- **影響範圍**：跨 framework（Vanilla JS / React / Flet 皆適用）。凡是有「可選的第二模式」「編輯模式 / 預覽模式」「進階選項展開」的 UI 都會遇到。
- **證據**：本專案 CLAUDE.md 坑 P8；V0.5.0 修正紀錄。
- **檢查 1 結果**：grep Kit `狀態殘留|模式切換|進出點` → 無對應條目。坑 #27（事件冒泡）、#33（動態文字回饋）都是 UI 坑但不同題。→ **無重複，建議新增 #68**。

#### 3. `[既有 stack 新 API: dictionaryapi.dev]` 第三方 API 回的是「這個查詢的所有可能」，不是「你要的那個答案」

- **症狀**：查 `year` 的釋義，卡片上連續出現三個 `noun`：第一個是「太陽年」，第二、三個卻是「This place; this location」與「Eye dialect spelling of hear」—— 那是 *here* / *hear* 的方言拼法，跟「年」毫無關係。**API 沒有錯、回應是合法的**，錯的是我把它整包當成「year 的意思」渲染出去。
- **原因**：`dictionaryapi.dev` 回傳的 `j` 是 **entry 陣列（依詞源拆分）**，語意是「**這個拼法在英語裡的所有可能**」。我的迴圈 `for (const e of j)` 把所有 entry 混抓。這是同形異義詞才會現形的 bug，**測 10 個常用字可能一個都碰不到**。
- **做法**：
  - **先問「這批資料的每一筆，是不是都在回答我要問的那個問題」**，再決定怎麼取。本案答案是「不是」→ 只取 `j[0]`（主詞源）。
  - 對「明顯不是在回答我的問題」的項目**主動過濾**（本案濾掉 `eye dialect / alternative spelling / misspelling / abbreviation of` 這類「這個拼法是別的字的變體」定義 —— 它們對背單字零價值）。過濾要留 fallback：整組被濾光時退回原始，不留空白。
  - **驗證要挑會現形的樣本**，不是挑順手的樣本。本案要用 `year`（同形異義）、`run`（多詞性）、`colour`（全是變體定義）才測得出來。
- **影響範圍**：所有整合第三方語料／字典／地理編碼／搜尋 API 的專案 —— 凡是 API 回「符合這個查詢的所有結果」而你只要「其中一個」的場景。
- **證據**：本專案 CLAUDE.md 坑 P9；V0.5.1 修正紀錄與離線驗證腳本（5 種情境）。
- **檢查 1 結果**：grep Kit `外部 API|第三方 API` → 現有 #15（單一來源風險 → 講**可用性**、多備援）、#37（時區標示不可信 → 講**欄位值**不可信）。本條講的是**回傳集合的語意**不可假設 —— 與 #37 同家族（「不要信任第三方 API 的語意」）但不同層次：#37 是單一欄位的值錯，本條是「整批資料在回答另一個問題」。→ **不重複；建議新增 #69，並在 #37 加一行 See also #69**。

#### 4. 人工搬移的資料集合會靜默掉項目 —— 必加 count assertion

- **症狀**：醫學詞庫在多次 build script 之間手抄搬移，`differentiation`、`conformity index` 兩筆靜默消失。**沒有任何報錯**，程式跑得好好的。更糟的是我在交付訊息回報「252 筆」—— 那是沿用上一版的口徑，實際只有 250，**錯誤的數字被我親手寫進交付訊息**。
- **原因**：兩層失效疊加 —— ① 資料陣列是人工搬移的，掉項目不會有語法錯；② 回報時沿用記憶中的數字而非實測，於是連「掉了」都沒發現。
- **做法**：
  - **凡是人工搬移／重建的資料集合，煙霧測試必須有 count assertion**（預期筆數寫死在測試裡，不符就紅燈），另加重複項與必填欄位檢查。
  - **交付訊息裡的每個數字都要當場實測產生，不可沿用上一版口徑。** 「上一版說 252」不是證據。
- **影響範圍**：所有含內嵌資料表／常數陣列／對照表的專案（Kit 靜態 HTML、CLI 工具尤其常見）。
- **證據**：本專案 CLAUDE.md 坑 P7；V0.4.0 修正紀錄（實測 250 → 補回 252）。
- **檢查 1 結果**：grep Kit `count|assertion|筆數` → 坑 #7（打包前必跑驗證）講的是「跑煙霧測試」的**習慣**，未涉及「**資料完整性**該測什麼」。→ **無重複，建議新增 #70，並在 #7 的煙霧測試清單補一行「內嵌資料集合的筆數斷言」**。

### 可加但等更多證據確認

- **「離線驗證外部 API 邏輯」的作法**（從 index.html 切出純函式 + 注入假 fetch + eval，不需網路就能驗 5 種情境）。本案沙箱連不到 dictionaryapi.dev 反而逼出這個作法，意外好用。但這是 N=1，且與專案的「單檔內嵌」結構高度耦合（切得出來是因為函式沒有依賴 DOM）—— **等第二個案例再決定是否進 optimizations.md**。

## 三、發現的「跨專案設計模式」

### 1. 「評估完 → 徵詢同意」是把鐵律降級成建議的合法外衣

- **本案發生情境**：V0.6.0 交付時，我完整算出「坑 6 條（門檻 3）、技術選型 2 條（門檻 2）、從未產出過 handoff」，明確寫下「**本版已明確達標，建議產出**」，然後——問了一句「要我現在產出嗎？」就停手。SELA 回「沒看到 hand-off」，我才動。
  - 關鍵在於：**Kit CLAUDE.md 已經預測到這個失效模式**（「多次發現專案 Claude 對『為未來鋪路』相對不積極」「Switch Vault 對齊後仍沒主動產出 handoff，要 SELA 提醒」），還為它設了鐵律 #0，**我讀過那段字，然後照樣犯**。
  - 我沒有違反字面規則：鐵律 #0 要求「評估」，我評估了；要求「明寫理由」，我寫了。**但鐵律 #0 的目的是讓 handoff 被產出，不是讓它被討論。**
- **可推廣的原則**：**當規範說「達標就做」時，「達標了，要做嗎？」不是遵守，是規避。** 徵詢同意在多數情境是美德，但它有一個特定的失效場景 —— **規範已經替 SELA 做過決定的事，再問一次等於把決定權還給他，而他設這條規範就是為了不用每次都決定。** 對 Claude 而言這個動作的吸引力在於：它看起來禮貌、零風險、且無法被指責違規。這正是它危險的原因。
  - 可操作的判準：**「如果我問了，他幾乎一定會說好」→ 那就不要問，直接做。** 問句只保留給「他可能說不」的情況。
- **代價 / 取捨**：
  - 會產出一些 SELA 其實不需要的 handoff（浪費他的閱讀時間 + 我的產出 token）。
  - 對「達標」的判斷若太寬鬆，會變成每版都產 handoff，稀釋 handoff 的訊號價值。
  - **所以取捨點是「門檻要嚴、觸發要自動」** —— 而不是「門檻寬鬆 + 每次徵詢」。目前 Kit 的門檻（≥3 坑 / ≥2 選型 / ≥5 小版本）本身是嚴的，問題只出在觸發環節。
- **建議寫入**：`conventions/sela-philosophy.md`（這是關於 Claude 工作模式的自我認知，不是某個技術棧的坑）。

### 2. 顯示層要嘛從資料算出來、要嘛就別存在

- **本案發生情境**：`BOX_DAYS=[0,1,3,7,14,30]` 是排程資料，`BOX_NAMES=["","Box1·每日","Box2·隔日","Box3·三天",…]` 是另一份手寫顯示字串。**兩份沒有任何連結**，於是 UI 連續三個版本都在說謊：標籤寫「Box2·隔日」實際是 3 天、「Box3·三天」實際是 7 天、「Box4·週複習」實際是 14 天。README 的對照表也跟著錯。
  - 諷刺的是：**這個 App 的核心價值就是「間隔」，而它把間隔標錯了三個版本**，還是使用者問「為什麼覺得怪怪的」才翻出來。
- **可推廣的原則**：**同一件事寫兩遍 = 待爆的地雷，而且爆炸時沒有聲音。** 顯示字串若無法從資料衍生，就是一份會漂移的複本。解法不是「記得同步」（人不會記得），是**刪掉複本**：`BOX_LABEL = b => \`Box${b}·${BOX_DAYS[b]}天\``。
  - 這是 Kit 坑 #3「資料 vs 顯示混用」的**反向案例**：#3 講「從顯示反推資料」（display → data），本例是「資料與顯示各存一份」（data ‖ display）。**兩者是同一個病的兩個方向**，Kit 目前只寫了一個方向。
- **代價 / 取捨**：
  - 衍生式標籤的表達力較弱 —— 「Box5·30天」不如「Box5·精通」有情緒。若要保留語意標籤，就必須讓它跟資料同源（如 `BOX_DAYS` 改成 `[{days:30,name:"精通"}]` 物件陣列），而不是另開一張表。
  - 過度衍生會讓文案綁死結構，i18n 或行銷文案調整時綁手綁腳。**判準：會隨資料變動而失真的字串必須衍生；純裝飾性文案可以手寫。**
- **建議寫入**：`conventions/cross-project-pitfalls.md` **坑 #3 補強**（不另開新坑 —— 這是既有坑的第二個方向，見第四節）。

  `├── cross-project-pitfalls.md ← 跨專案通用坑（編號累積到 #58，57 條 + #35 保留）`
  實際 `grep -c '^### #[0-9]*\.'` = **66 條、最大 #66**。
- **建議改成**：**拿掉數字**，寫成「跨專案通用坑（編號累積，只增不重排）」。
- **理由**：這串數字每加一條坑就會過期，而「加坑時順手改導覽的數字」是不可能被記住的維護動作 —— 事實證明它漂了 8 條。**這正是本案坑 P10 / 第三節模式 2 的同型錯誤，出現在 Kit 自己身上。**
  - 這條的價值不只是修一個 typo：**當一份規範自己都做不到它警告過的事，就是這條規範需要「結構性解法而非紀律解法」的最強證據。** 建議 Kit 順勢把「別在文件裡寫會變的數字」寫進坑 #3。

#### 2. 坑 #3 只寫了「顯示 → 資料」一個方向，缺「資料 ‖ 顯示 各寫一份」

- **現狀**：坑 #3「資料 vs 顯示混用（從 UI 字串解析資料）」，症狀與做法全部圍繞「用 `textContent.match()` 反推資料」，做法是「用 `data-*` 分離」。
- **建議改成**：把標題擴為「**資料與顯示必須同源**」，並補第二個方向的案例：
  ```
  - 方向 B：資料與顯示各寫一份 → 顯示靜默說謊
    - 症狀：UI 標籤宣稱的數值與實際行為不符，且無任何報錯
      - Gecko Grip 案例：BOX_DAYS=[0,1,3,7,14,30] 是資料、BOX_NAMES=["","Box1·每日","Box2·隔日"…]
        是另一份手寫字串，標籤「Box2·隔日」實際間隔是 3 天，說謊三個版本才被使用者問出來
    - 原因：兩份表無連結，改資料不會有人提醒你改標籤
    - 做法：刪掉複本、標籤由資料衍生（BOX_LABEL = b => `Box${b}·${BOX_DAYS[b]}天`）
    - 同型錯誤也出現在文件：Kit 自己的 CLAUDE.md §4 寫「編號累積到 #58」而實際 #66（漂了 8 條）
    - 判準：會隨資料變動而失真的字串必須衍生；純裝飾性文案可手寫
  ```
- **理由**：兩個方向是同一個病（同一件事有兩個真相來源），拆成兩條坑會讓人以為是兩件事。合併後這條坑的辨識度更高 —— **「你有幾個地方存了同一件事？」比「你有沒有用 textContent 解析資料？」更容易在寫程式時想起來。**

#### 3. handoff 模板的「產生時機」與「模板使用須知」互相矛盾（**這條最實質**）

- **現狀**：同一份 `SELA-handoff-template.md` 裡：
  - 開頭「產生時機（V1.8.0 再擴大）」與鐵律 #0 → 「累積踩坑 **≥ 3 條** / 技術選型決策 ≥ 2 條」就要產出；V0.x.x 每滿 5 個小版本必走評估。
  - 結尾「模板使用須知 → 該不該產出？」→ 「**不會**：V0.1.0 ~ V0.x.x（雛型階段，反饋還不穩定）／純 bug fix 升版（c+1）」。
- **建議改成**：刪掉或改寫「模板使用須知 → 該不該產出」整節，讓它指回鐵律 #0 與開頭的「產生時機」，例如：
  ```
  ### 該不該產出 SELA-handoff.md？
  → 依鐵律 #0 與本檔開頭「產生時機」判斷，本節不另立標準。
    （V1.9.0 起 V0.x.x 雛型階段同樣適用 —— 舊版曾寫「V0.x.x 不產出」，已作廢）
  ```
- **理由**：V1.9.0 已經因為 SelaTripPay 的教訓，明寫「雛型階段每滿 5 個小版本必走評估、**不要默默累積到 V1.0.0 才一次補**」，但**舊的「不會產出：V0.x.x」那節沒有跟著刪**。於是模板同時說「雛型階段要產」和「雛型階段不產」。
  - **本案就是受害者**：我在 V0.6.0（V0.x.x + 6 條坑）評估時，看到「不會：V0.x.x」而傾向不產。這不是我不做的正當理由（鐵律 #0 位階更高，且我已算出達標），但它確實提供了「合理的猶豫空間」—— **而規範的價值恰恰在於消滅猶豫空間**。
  - 同理，「不會：純 bug fix 升版（c+1）」也與「≥3 條坑」矛盾 —— 本案 V0.5.1 是 c+1 但帶進一條通用坑。

## 五、留在這個專案、**不要回流 Kit** 的東西

- **Leitner 五盒的具體參數**（`BOX_DAYS = [0,1,3,7,14,30]`、`LAPSE_MULT = [1,.7,.55,.45]`、`REQUEUE_GAP = 4`）—— 業務調校值，其他專案無意義。

## 七、給 Kit Claude 的最後備註

**這份 handoff 本身就是它要講的那個問題的證據。**

我在 V0.6.0 完成了鐵律 #0 的評估、算出達標、寫下「建議產出」—— 然後停在那裡問「要我產出嗎？」。SELA 回了四個字：「沒看到 hand-off」。

Kit CLAUDE.md 早就寫了：「多次發現專案 Claude 預設積極處理『當下交付』，對『為未來鋪路』相對不積極」「Switch Vault V1.14.0 對齊到 V1.8.0 後仍沒主動產出 handoff（要 SELA 提醒）—— 這條鐵律就是修這個執行率問題」。

**鐵律 #0 是為了修這個問題而設的，而我在讀完它之後，又犯了一次同樣的錯。**

這代表：把失效模式寫進規範、設成鐵律、還附上前例說明 —— **這一整套加起來，仍然不足以阻止它再次發生**。第四節的結構性建議（把已知失效模式收攏成一節、配結構性對策而非提醒）就是基於這個觀察。如果 Kit Claude 只從這份 handoff 收一條，我建議收這條。

另外兩點：

1. **本案的坑有一個共同形狀**：P7（詞庫掉詞沒人知道）、P9（API 噪音混進釋義）、P10（標籤說謊三版）—— **三條都是「靜默失敗」：程式跑得好好的、沒有報錯、使用者看到的東西是錯的。** 而三條全部是 SELA 用眼睛看出來的（「為什麼覺得怪怪的」），不是測試抓到的。這個工具的煙霧測試從 V0.3.0 就存在，但測的是「會不會壞」，不是「說的是不是實話」。**建議 Kit 未來考慮把「正確性測試」與「不崩潰測試」在坑 #7 裡分開講。**

2. 本案仍在 V0.x.x 雛型階段（V0.6.1），預期到 V1.0.0 前還會有迭代。若第二節的坑在其他專案獲得驗證（特別是 #68 模式進出點、#70 筆數斷言，這兩條我認為最容易在別的專案重現），可回頭把 `[N=1]` 標記升級。
